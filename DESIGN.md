## 0. これは何か

スマホ主体のアナログ（紙）カードゲーム対戦を支援する**汎用ツール**。
「その場で使うカウンター類」と「対戦記録・分析」を**1つに統合**し、
カウンター操作の結果（先後・マリガン・ライフ推移）が**そのまま記録に自動で流れ込む**のが最大の差別化点。

### 絶対条件（ユーザー要件）
- モバイル主体。PCとも同期して気軽にデータ管理。
- 友人に渡して使える（配布が簡単）。
- **ランニングコスト完全に0円**（API・サーバー課金なし）。
- できる限り選択式入力で手間を減らし、"使っていて気持ちいい"工夫。

### ユーザーのプレイ環境（初期プリセットの根拠）
- **メイン: MTG**（EDH/統率者、レガシー等 多フォーマット）。→ ライフ20と40の両方、d20多用、多人数EDH、統率者ダメージが絡む。
- **次点: ホロカ（ホロライブOCG）**。ライフ管理は本体で完結しがちなので、汎用機能で十分の想定。
- 汎用ツールなので、**他ゲームはユーザーが後から手動プロファイル追加**できる設計にする。

## 1. 技術構成（確定）

| 項目 | 決定 | 理由 |
|---|---|---|
| 形態 | **PWA**（インストール不要・ホーム画面追加・オフライン可） | ストア審査不要、友達にはURL配布のみ、電波の悪い会場でも動く |
| 保存 | **端末内 IndexedDB**（Dexie.js 経由） | サーバー不要＝通信もサーバー代も0。ローカルファースト |
| 同期 | **手動ファイル書き出し/読み込み（JSON全体＋CSV分析用）** | ランニングコスト0。将来の自動同期は"足せる"設計にしておく（下記） |
| ホスティング | **GitHub Pages / Cloudflare Pages（無料枠）** | 維持費0円。まずはローカルVite devで開発 |
| 言語/FW | **TypeScript + React + Vite** | 最も定番でClaude実装の質が安定。カウンターの状態管理に向く |
| スタイル | **Tailwind CSS** | 素早くモバイル最適UIを組める |
| PWA化 | **vite-plugin-pwa**（Workboxベース） | オフライン・ホーム追加を設定ファイルで完結 |
| グラフ | 軽量ライブラリ（例: uPlot か Recharts）※フェーズ3で選定 | ライフ推移・勝率の可視化 |

### 同期方針の詳細（重要・ユーザーへの案内込み）
- 実装は「**全データをJSONで書き出し / 読み込み**」＋「**記録をCSVで書き出し**（分析用・読み込み不要）」。
- **推奨運用**: 書き出し先をGoogleドライブ等のPC同期フォルダにするだけで、体感ほぼ自動同期。コード追加ゼロ・0円のまま "PC同期" を満たせる。ユーザーにはこの運用をまず案内する。
- **将来の自動化の逃げ道**: データ層を `SyncAdapter` インターフェース越しに触る設計にしておき、後日 Firebase無料枠 等の実装を差し込めるようにする（今は作らない）。

## 2. データモデル（実装の背骨）

IndexedDB（Dexie）。ID は UUID 文字列。日時は ISO8601 文字列。

```ts
// ゲームのプリセット定義（ユーザー追加も可能）
GameProfile {
  id: string
  name: string                 // "MTG (Legacy/1v1)", "MTG (EDH/統率者)", "ホロカ"
  startingLife: number         // 20 / 40 / (ホロカは概念調整)
  playerCountDefault: number   // 2 / 4
  supportsCommanderDamage: bool // EDH=true
  counterTypes: CounterType[]  // 毒/経験/エネルギー/統率者ダメージ 等（ゲーム別）
  diceDefaults: DieType[]      // よく使うダイス
  isBuiltIn: bool              // 内蔵プリセットか、ユーザー追加か
}

CounterType { key: string; label: string; defaultValue: number; icon?: string }
DieType = 'd4'|'d6'|'d8'|'d10'|'d12'|'d20'|'d100'

// 自分のデッキ（対面デッキも同じ概念で候補サジェストに使う）
Deck {
  id: string
  gameProfileId: string
  name: string
  isMine: bool                 // 自分のデッキ / 対面として登録した相手デッキ
  archetypeTag?: string        // 分析用の大分類（任意）
  createdAt: string
}

// 1マッチ = 1回の対戦単位（BO1でもBO3でも1マッチ）
Match {
  id: string
  gameProfileId: string
  date: string                 // 対戦日
  tags: string[]               // 大会名・フォーマット・自由タグ（分析軸）
  myDeckId?: string
  opponentDeckIds: string[]    // 1v1は1つ、EDHは複数
  result: 'win'|'lose'|'draw'  // マッチ全体の結果（BO3は勝ち越しで確定 or 手動）
  format?: 'BO1'|'BO3'|'BO5'|'EDH-pod'
  note: string                 // 自由入力備考
  createdAt: string
}

// マッチ内の各ゲーム（BO3なら最大3つ）。ここに自動記録データがぶら下がる
GameRecord {
  id: string
  matchId: string
  gameIndex: number            // 1,2,3...
  onThePlay?: 'me'|'opponent'  // 先手/後手（ダイス/コイン結果と同期して自動記録）
  mulligans: { me: number; opponent?: number } // マリガン数（カウンターと同期で自動記録）
  result?: 'win'|'lose'|'draw'
  lifeLog?: LifeEvent[]        // ライフ推移ログ（カウンター操作を裏で自動記録）
}

LifeEvent { t: string; player: string; from: number; to: number; reason?: string }
```

設計メモ:
- **ライフ推移ログは"カウンター操作を保存するだけ"なので技術的に軽い**（ユーザーの懸念どおり煩雑にならない）。ライブ画面のライフ増減イベントを `LifeEvent` として貯め、記録保存時に `GameRecord.lifeLog` へ移すだけ。
- **対面デッキは Deck(isMine=false) として蓄積**し、次回入力で候補サジェスト → 選択式で手間削減。
- 勝率などの分析は Match/GameRecord を集計して算出（保存しない、常に再計算）。

## 3. 機能仕様（フェーズ別）

### フェーズ0 — 基盤
- Viteプロジェクト作成（TS+React+Tailwind+Dexie+vite-plugin-pwa）。
- Dexieスキーマ定義、上記データモデル実装。
- **内蔵GameProfileのシード**: MTG(Legacy/1v1, life20)、MTG(EDH, life40, 統率者ダメージON, 4人)、ホロカ（暫定）。
- ホーム画面（ゲームプロファイル選択 → ライブ画面 or 記録一覧へ）。ボトムタブ: [対戦(ライブ)] [記録] [設定]。

### フェーズ1 — カウンター系（"すぐ触れて気持ちいい"を最優先）
- **ライフカウンター**
  - 開始値プリセット（プロファイル由来）ワンタップ。
  - 大きい ± ボタン、**ロングタップで ±5/±10 の速い増減**、直接数値タップで手入力。
  - **Undo（直前操作を戻す）**。
  - **1v1モード / 多人数モード切替**。多人数は各プレイヤーのライフを並べる。
  - **統率者ダメージ**（EDH）は各対戦相手からの被ダメをカウンターの一種として保持。
  - 全増減を裏で `LifeEvent` として記録（フェーズ2で記録に接続）。
- **コイントス**: タップでアニメ→表裏。ホロカ/ポケカ的な連続トスにも配慮（回数指定 or 連打）。
- **ダイスロール**: d4/d6/d8/d10/d12/d20/d100 フルセット。個数指定、合計表示、**履歴**。
- **マリガンカウンター**: 自分（と任意で相手）のマリガン回数を+/-。ライブ画面から即記録に渡せる。
- **先後決めフロー**: 「ダイス/コインを振る → 勝った側が先攻/後攻を選ぶ」を1つの流れにして、結果を `onThePlay` として保持。

### フェーズ2 — 記録連携
- ライブ画面から「この対戦を記録する」→ 先後・マリガン・ライフ推移が**自動で埋まった状態**で記録入力へ。
- 記録入力画面（選択式中心）: 日付（既定=今日）、タグ（過去タグから選択＋新規）、自分デッキ（選択）、対面デッキ（候補サジェスト＋新規）、勝敗（大ボタン）、備考（自由入力）。
- BO3対応: マッチ内に複数 GameRecord をぶら下げ、勝ち越しで Match.result 自動判定。

### フェーズ3 — 履歴・分析
- 記録一覧（新しい順）。**ソート/フィルタ: 対面デッキ別・日付・大会/タグ別**。
- 集計: 全体勝率、対面デッキ別勝率、先手/後手別勝率、マリガン数と勝敗の相関、期間推移。
- **CSV書き出し**（スプレッドシートで自由分析）。フェーズ3で簡単なグラフも。

### フェーズ4 — 配布・同期
- vite-plugin-pwa 設定（manifest, オフラインキャッシュ, アイコン）。
- JSON全体の書き出し/読み込み（`SyncAdapter` 経由）。
- GitHub/Cloudflare Pages へデプロイ → URL配布。ユーザーにクラウドフォルダ運用を案内。

### フェーズ5 — 拡張
- EDH多人数の記録構造（ポッド、複数対面、統率者ダメージの記録）。
- ゲームプロファイルのユーザー追加UI。追加ゲーム対応。

## 4. "気持ちいい" UX 原則（craft指針）
- 入力は**選択・サジェスト・大ボタン優先**、キーボードは最終手段。
- 会場で片手・親指操作を想定。主要操作は画面下側に。
- Undo を随所に。誤タップに寛容。
- ダークモード標準（対戦卓は暗いことが多い、電池も持つ）。
- 画面遷移は浅く。ライブ↔記録の往復が最短。

---

## 5. 非目標 / 後回し（スコープ管理）
- オンライン対戦・リアルタイム共有はしない。
- カード画像DB・価格・デッキ構築支援はしない（対戦支援に集中）。
- アカウント/ログインなし（ローカル保存が基本）。
- 自動クラウド同期は今作らない（差し込める余地だけ残す）。

### 設計から進化した点（実装者は必ず把握すること）
- **編集UI**: デッキ/タグ/プレイヤーの編集は「別画面のリスト管理」ではなく**選択チップ自体が編集モードに切り替わる方式**（dnd-kit採用）。
- **内部の紐付けが2種類ある（要注意）**: **タグ＝名前カスケード方式**／**デッキ・プレイヤー＝ID参照方式**。改修時にこの違いを取り違えないこと。
- **UI設定の保存**: 表示項目on/off・簡易表示などのUI設定は **localStorage + React Context**（Dexie不使用）。対戦/記録データのDexieとは別レイヤー。
- **EDH（多人数）**: 画面が狭くなるため、ユーザー設定に関わらず**簡易表示を強制デフォルト化**。
- **対面プレイUI**: 180度回転・向き切替(⇅)対応済み。
- **記録連携**: BO1/3/5の勝敗自動判定、連戦（連続対戦）ボタン実装済み。
- **分析**: 勝率・デッキ別・先手後手別・マリガン別・累積勝率グラフ（自作）＋CSV書き出し。

### 保留/既知の制約

## 8. ポリッシュフェーズ仕様（統括確定 2026-07-08 / 次セッション実装対象）

### 8-1. 見た目ブラッシュアップ
**方向性＝「上質ミニマル・ダーク」を土台に、ゲーム別テーマをレイヤーで乗せる。**
- **土台**: 深いダーク背景、アクセントは基本1色、余白・階層・タイポグラフィを整えて可読性と高級感を両立。卓で数字がパッと読めること・電池優先を最上位に。既存UIを壊さず質を上げる方向（機能は完成済み、見た目の底上げが目的）。
- **ゲーム別テーマ（採用範囲＝アクセント色＋地の色＋ユーザー壁紙）**:
  1. **アクセント色**: GameProfileに accentColor を追加。ボタン強調・グラフ線・選択状態等に反映。
  2. **地の色**: プロファイルごとに背景に淡い色味/グラデ（暗いまま＝可読性・電池を損なわない範囲）。
  3. **ユーザー壁紙**: ユーザーが自分で画像を選び、ゲーム別背景に設定できる機能。
     - 画像は **IndexedDB に blob 保存**（プロファイルID紐付け）。バックアップJSONの扱いは要検討（サイズ大なので既定は含めない or 任意）。
     - **既定オフ・無地ダーク**。設定した時のみ適用。
     - 画像の上に **自動でスクリム（半透明の暗幕）** を重ね、数字/テキストの可読性を担保。compactLive/EDH時は自動で更に簡素化 or 壁紙抑制。
     - **内蔵アート素材は入れない**（公式カードアートは著作権NG、容量も回避）。ユーザー持ち込み画像＝自己責任で自由、が設計の肝。

### 8-2. Wake Lock（画面オフ防止）
- **Screen Wake Lock API** を使い、対戦（ライブ）画面表示中に画面が自動で消えないようにする機能。
- **設定にトグルを用意、既定＝オフ**（端末任せ）。ONの時だけ wake lock を取得、画面/タブ非表示で解放、復帰で再取得。
- 電池増のトレードオフである旨を設定画面に一言添える。

### 8-3. バッテリー（実装は監査のみ、主因は画面点灯時間）
- **常時ループの監査**: setInterval / requestAnimationFrame が対戦中に回りっぱなしになっていないか確認し、あれば停止/最小化。コイン・ダイス等の一時アニメは可。
- ダークテーマ維持（OLED実消費に効く）。それ以上のコード最適化は不要（バックグラウンド消費は元々ほぼゼロ＝通信/プッシュ/常駐なし）。

### 8-4. その他
- **MITライセンス追加**（統括承認済み）。LICENSEファイル＋必要ならpackage.json。
- （任意）久しく未バックアップ時にそっと促す一言。

## 9. ポリッシュフェーズ実装済み（as-built / 2026-07-08 追記）

§8の内容はすべて実装・型チェック・実機（preview）確認まで完了。

- **アクセント色**: `GameProfile.accentColor`（任意, hex）を追加。内蔵3プロファイルに設定（MTG 1v1=sky、EDH=violet、ホロカ=pink）。
  `App.tsx` が選択中プロファイルの色をCSS変数 `--accent` としてアプリ全体に配布し、各所で `bg-[var(--accent)]` 等の形で参照する
  （ボトムタブの選択状態、「対戦を記録する」ボタン、ダイスの選択/実行ボタン、デッキ選択チップの既定色、フォーマット選択、
  勝率トレンドグラフの線、Sheetモーダルの上辺ライン）。タグ/対面プレイヤーチップの緑・アンバーは意味的な色分けとして維持
  （アクセント色を被せていない）。
  **設定タブは意図的にアクセント色を使わない**（統括フィードバック：スイッチ/文字の色が場面によって変わるとかえって
  混乱するため、常に固定色）。
- **地の色**: 二重管理を避けるため、プロファイルごとの背景色フィールドは持たせず、アクセント色から薄い放射グラデーション
  （`hexToRgba` で計算）を対戦（ライブ）画面の背景として自動生成する方式にした（`src/lib/color.ts`）。
- **ユーザー壁紙**: Dexieに `wallpapers` テーブル追加（v4, 主キー=id文字列, blobで保存）。
  `src/db/wallpaper.ts` で get/set/clear。idは2種類使い分ける：ゲームごとの壁紙は `gameProfileId`、
  ホーム画面専用の壁紙は固定キー `HOME_WALLPAPER_ID`（`'__home__'`）。`src/live/useWallpaperUrl.ts` でobjectURL化、
  `src/screens/WallpaperSection.tsx` を汎用部品化（`id`/`title`/`description`を渡すだけで両方に使い回せる）。
  **壁紙は2箇所、それぞれ独立して設定できる**（統括フィードバックにより試行錯誤の末この形に確定）：
  1. **ホーム画面の壁紙**（設定タブ→「ホーム画面の壁紙」、`HOME_WALLPAPER_ID`）: 「対戦」タブ最初の
     『ゲームを選ぶ』一覧ページ全体の背景に使う（`LiveScreen.tsx` の `!profile` 分岐、`homeBgStyle`）。
     ゲームがまだ選ばれていない状態なので、どのゲームにも属さない汎用の1枚。
  2. **ゲームごとの壁紙**（設定タブ→ゲームを選んで「このゲームの壁紙」、`gameProfileId`）: そのゲームの対戦
     （ライブ）画面の背景に使う。多人数(EDH)/文字を減らす設定のときは可読性優先で暗幕を濃くする。
  どちらも暗幕（スクリム）を重ね、`background-position: center top` で縦長の人物イラストの顔が
  切れないよう上寄りを基準に切り抜く。ゲーム選択カード自体には壁紙を乗せない（試したが小さすぎて
  絵がほぼ見えず不採用）。バックアップJSON（backup.ts）には含めない。
- **適用範囲の実装判断**: アクセント色はアプリ全体（設定タブを除く）、地の色は対戦（ライブ）画面限定、
  壁紙写真はホーム画面全体＋各ゲームの対戦画面全体（どちらも独立設定）、という使い分けにした。
- **Wake Lock**: `src/lib/useWakeLock.ts`。`UiPrefs.wakeLockEnabled`（既定false、localStorage永続）を追加し、
  設定タブ「画面・電池」にトグル＋バッテリー注意書きを表示。対戦タブを実際に見ている間だけ有効化
  （`isActiveTab` をApp→LiveScreenに渡して判定）。タブ非表示→復帰で自動再取得。API非対応環境では
  トグルの代わりに非対応メッセージを表示。

  アンマウント時に確実に`clearInterval`される設計を確認済み（追加の停止処理は不要）。他はコイン/ダイス/先後演出・
  トースト表示用の一回限りの`setTimeout`のみで、常時回り続けるループは存在しない。

## 10. ポリッシュフェーズ以降の追加実装（as-built / 2026-07-09追記）

### 10-1. 未バックアップ時のそっと促す一言
`src/lib/backupReminder.ts` にlocalStorageで「最後に書き出した日時」を記録。`exportBackup()`実行時に自動記録。
設定タブのバックアップ欄で、**対戦記録が1件以上あり**、かつ「一度も書き出していない」または「前回から14日以上」の
時だけ、amber色の控えめな一言を表示する（`REMINDER_THRESHOLD_DAYS = 14`）。データが無いうちは急かさない。

### 10-2. 誤タップ対策（対面プレイで相手側ボタンがブラウザ上部バーに近い問題）
- **土台**: 対戦画面・ホーム画面の上端に `max(0.75-1rem, env(safe-area-inset-top))` の余白を追加。ノッチ端末にも対応。
- **本質的な解決**: `src/lib/useFullscreen.ts`。設定タブ「画面・電池」に**全画面表示トグル**（既定オフ、
  `UiPrefs.fullscreenOnPlay`）を追加。ONの時、ゲームを選んだタップの勢い（ユーザー操作の中）でFullscreen APIを
  呼び出し、ブラウザの上部バーごと隠す。ブラウザの仕様上、全画面化は必ずユーザー操作のハンドラ内から呼ぶ必要が
  あるため、「ゲームを選ぶ」ボタンのonClickに相乗りさせている。「ゲームを選び直す」で全画面を解除。
  API非対応環境ではトグルの代わりに非対応メッセージを表示（iOS Safariは基本非対応）。

### 10-3. ライフON/OFF・初期値編集・自由カウンター（統括承認済み設計）
検討の結果、「全カウンターを自由配置」ではなく、**ライフは特別扱いのまま維持しつつON/OFF可能にし、
その他カウンターは統率者ダメージのUIパターンを流用**する折衷案を採用（作り込んだライフUIの質を保ちつつ、
ホロカのようにライフ不要なゲームにも対応する）。

- `GameProfile.hasLifeCounter?: boolean`（未設定はtrue扱い＝既存プロファイルと後方互換）。falseなら対戦画面から
  ライフの大きな数字・±ボタン・クイックチップが丸ごと消える。
- `GameProfile.counterTypes` が**装飾目的から実際に機能するカウンターへ昇格**。`PlayerState.counters: Record<string, number>`
  を`useBoard.ts`に追加し、`changeCounter(playerId, key, delta)`で+/-（下限0でクランプ、lifeLogのような詳細履歴は
  持たせず「不足分はフィードバック待ち」の方針で軽く実装）。`LifeCounter.tsx`に統率者ダメージのすぐ下へ
  「その他カウンター」チップ行として表示。
- **ホロカの扱い**（統括判断）: エール/ホロパワーは外部カウント不要とのことで`counterTypes: []`に変更。
  一方ライフは、対戦中に増減させる場面がなくても**勝敗確定時の最終値を記録・分析したい人がいるかもしれない**
  という理由で`hasLifeCounter: true`のまま維持（手入力で最終値を入れる用途）。
- `hasLifeCounter: false`のプロファイルでは、`GameResultForm`のライフ行を非表示にし（`showLife`prop）、
  `GameRecord.finalLife`も保存しない（無意味な0を記録しないため）。
- MTG(1v1/EDH)の毒・エネルギー・経験は、機能を作った直後は実際に押せるカウンターにしてみたが、
  「使わない時に邪魔」というフィードバックにより**内蔵ゲームでは既定オフ（`counterTypes: []`）に戻した**
  （2026-07-10）。その他カウンターの仕組み自体はそのまま残っており、必要な人は「ゲームを追加」で
  自分用プロファイルに自由に足せる。

### 10-4. ゲームプロファイルのユーザー追加/削除（DESIGN.md §3フェーズ5の残タスク）
- `src/db/profiles.ts`: `createGameProfile()` / `deleteGameProfile()`。削除は**統括承認済みの方針**で、
  確認ダイアログで警告した上でデッキ・タグ・プレイヤー・対戦記録・壁紙を含めて完全にカスケード削除する
  （「取り消し」機能は持たせず、復元したい場合はバックアップの読み込みに委ねる）。
- `src/screens/GameProfileForm.tsx`: 新規ゲーム作成フォーム（名前・人数・ライフON/OFF+初期値・統率者ダメージ・
  自由カウンター追加・ダイス既定・アクセント色）。カウンター種類は**自由入力のみ**（毒/エネルギーのような
  ゲーム別の細かい対応は用意せず、不足はフィードバック待ちの方針）。
- 設定タブのプロファイル一覧に「+ ゲームを追加」ボタン（Sheetでフォームを開く）と、ユーザー追加分のみに
  削除ボタンを表示（`isBuiltIn`で判定、内蔵プロファイルは削除不可）。
- `DieType`の全種類配列を`db/types.ts`の`ALL_DICE_TYPES`に集約し、`DiceRoller.tsx`と`GameProfileForm.tsx`で共有。
- `PrefToggle`を`src/components/PrefToggle.tsx`に切り出し、Settings画面とプロファイル追加フォームの循環参照を回避。

### 10-5. 復元データの後方互換クラッシュ修正
`PlayerState`に`counters`フィールドを追加した際、スマホのlocalStorageに残っていた**旧バージョンの
保存済み対戦データ**（`counters`が無い）を復元すると、その他カウンター欄で`undefined`を参照してJS例外
→ 画面が真っ黒になる不具合が発生（実機で発見）。`useBoard.ts`に`normalizeBoard()`を追加し、復元のたびに
欠けているフィールド（`commanderDamage`・`counters`）を補ってから使うようにした。今後フィールドを追加する
際も同様の後方互換クラッシュに注意すること。

### 10-6. 対面回転パネルでのカウンター折り返し重なり
統率者ダメージ／その他カウンターの行が`flex-wrap`で2行に折り返わると、180度回転した相手側パネルでは
行の上下・左右が反転するため、2行が視覚的にぶつかって重なって見える不具合が実機で見つかった。
`flex-nowrap overflow-x-auto`（各チップは`shrink-0`）に変更し、折り返さず必要なら横スクロールする形に修正。
**回転させるパネル内のリストは今後も折り返しではなく横スクロールを基本にすること。**

### 10-7. 全画面表示の技術的な限界（統括への回答）
Fullscreen APIは仕様上「ユーザー操作の中からしか呼べない」ため、ページ再読み込みで対戦セッションを
自動復元した場合は発動しない（自動全画面化はブラウザの安全装置により不可能。全画面はいつでもシステム操作で
抜けられる必要があり、ページ側で塞ぐこともできない）。対策として対戦画面に「タップして全画面表示にする」
ボタンを追加し、ワンタップで復帰できるようにした。
なお「ホーム画面のアイコンから開いた時にブラウザ感を消したい」という本来の目的には、Fullscreen APIより
**PWAのstandaloneインストール**（`vite.config.ts`の`display: 'standalone'`）の方が本筋の解決策で、
正しくインストールされていれば自動的にバーが出ない。全画面トグルは、通常のブラウザタブとして使う場合の
保険という位置づけ。

### 10-8. 使い方・よくある質問のアプリ内包
- `src/content/helpContent.ts`: カテゴリ別Q&Aのデータ（基本の使い方／対戦の記録／カスタマイズ／画面・電池／
  データ・バックアップ／よくある質問・トラブルの6カテゴリ、計30項目）。画像なし・外部通信なしでバンドルに
  埋め込むだけなので、容量・バッテリーへの影響はほぼ無視できる（gzip後で+4KB程度）。
- `src/screens/HelpSection.tsx`: カテゴリごとに見出し、各質問はタップで開閉するアコーディオン。
- 設定タブの一番上（プロファイル一覧より前）に「使い方・よくある質問」ボタンを配置し、Sheetで開く。
- **運用ルール（継続適用）**: 新しい機能を作ったら`helpContent.ts`のQ&Aも一緒に追記・更新すること。
  友人にフィードバックを求める前段階として、このドキュメント作成が実装された（2026-07-10）。
